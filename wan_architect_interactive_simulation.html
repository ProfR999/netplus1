SME 3-Site Architectural Insight

The "Under-the-Hood" Routing Dynamics:

In a 3-site network, legacy technologies (like T1 point-to-point lines) require a Hub-and-Spoke topology. This means the Branch cannot talk directly to the Cloud DC. It must route its traffic *through* the HQ Hub, adding latency overhead. Conversely, MPLS and SD-WAN dynamically enable direct mesh routes, reducing hops.

The Tromboning (Hairpinning) Pitfall

What is Hairpinning? Students often assume that if a branch has standard internet access, it can automatically route directly to cloud nodes. Under legacy enterprise VPN settings (split-tunneling disabled), all remote branch cloud requests are sent through the corporate HQ firewall first, then shot back out to the internet to reach the Cloud. This redundant double-back path mimics a trombone, wasting HQ bandwidth and doubling regional latency!

Technical Glossary (3-Site Model)
  • Full Mesh: Every node connects directly to every other node without mid-hops.
  • Hub & Spoke: Spoke nodes route exclusively via a central hub, creating latency bottlenecks.
  • Split Tunneling: Direct-to-internet routing for cloud apps from spokes, bypassing the HQ firewall.
Architectural Troubleshooting

Always calculate site-to-site latency. Even if Node A is connected to Node B and Node B is connected to Node C, routing traffic from A to C via B will double latency. Look for Direct-Mesh configurations (like MPLS or SD-WAN dynamic overlays) when latency is critical.
